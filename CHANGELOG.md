# Changelog

## [Unreleased]

A long-overdue modernization pass on top of the original 2015 codebase. Public API surface still uses `wasy.Wasy`; several internal classes have been replaced with factory functions or interfaces.

### Added

- **`SmfPlayer.audibleTick` / `Wasy.audibleTick`** — the tick position currently audible at the speakers, as opposed to `timer.tick`, which is the _scheduling frontier_ and leads the sound by the lookahead. Computed as `timer.tick + (currentTime − timer.currentTime − lookaheadSeconds) × ticksPerSecond`, clamped to `[0, timer.tick]`; returns the frontier as-is while paused. `examples/seekable-player` now tracks the audible state everywhere: the piano-roll now line, seek bar, and time/BPM/bar:beat readouts use `audibleTick`, and `onTimedEvent` view updates (keyboard lights, channel status, event log, marker sync) are queued until `audioContext.currentTime` reaches each event's scheduled audio time. Previously every visual led the sound by the whole lookahead — at 500 ms, note releases/decays visibly "ended" half a second before they stopped sounding.
- **`SmfPlayer.lookaheadSeconds` / `Wasy.lookaheadSeconds`** — the scheduling lookahead (`Timer.delayInSeconds`, default 200 ms) is now a first-class, clamped property (50 ms – 1 s; `MIN_LOOKAHEAD_SECONDS` / `MAX_LOOKAHEAD_SECONDS` exported). Events are dispatched this far ahead of the audio clock, so dispatch-driven UI and `timer.tick` lead the audible sound by the same amount; the documented contract is to change it **while stopped**, since already-dispatched events keep their old offset. `examples/seekable-player` gains a **Preferences panel** (hidden by default, toggled from the view bar) with a lookahead slider (50–500 ms) that is locked during playback, persisted to `localStorage`, and re-applied when the player is created.
- **Variable-duty pulse waves (`OscillatorSource.duty`).** Web Audio's built-in `"square"` is fixed at 50% duty; the new `src/synth/periodic-wave.ts` builds band-limited rectangular waves of arbitrary duty from the Fourier series (`real[n] = (2/πn)·sin(2πnd)`, `imag[n] = (2/πn)·(1−cos(2πnd))`, 64 harmonics) and caches one `PeriodicWave` per `(AudioContext, duty)`. `ToneDefinition` oscillator sources accept an optional `duty` (only meaningful with `oscillatorType: "square"`); `compileTone` forwards it to `SimpleOscillatorPatch.duty` and `onNoteOn` plays the pulse via `setPeriodicWave` instead of setting `type`. This unlocks the NES 2A03 palette: 12.5% (thin / nasal), 25% (the classic chip lead), 50% (hollow, unchanged default). **The entire `gmPatches` set was reviewed against the new capability** — pianos moved to 25% pulses (12.5% for Bright Piano), harpsichord / clav / muted guitar / muted trumpet / oboe / soprano sax / banjo / shanai / woodblock / Pad 6 (metallic) / FX 1 (rain) went thin at 12.5%, brass / saxes / reed organ / harmonica / picked & slap & synth bass / steel guitar / chiff lead / Fantasia pad / shamisen / synth drum / telephone sit at 25%, and clarinet / Lead 1 (square) / percussive organ keep the physically-right plain 50% square. `gmDrumKit`'s cowbell (25%) and claves / wood blocks (12.5%) thinned out likewise; Bass Drum 1 keeps 50% for low-end weight.
- **`examples/patch-editor/`** — interactive patch designer. Two modes: **Melody** edits a single `ToneDefinition` (oscillator/noise source; ADSR / ramp envelope with optional one-shot) and **Drum** edits a `DrumKitDefinition` (per-voice GM key 35–81; source, envelope, routing, exclude group). An interactive piano keyboard at the bottom triggers the current patch — click to play, or select a Web MIDI input port from the header. The **JSON** panel on the left live-serializes the definition; hand-edits are accepted on **Apply**; **Copy** puts the result on the clipboard for use as a `ToneDefinition` / `DrumKitDefinition` literal. Uses `compileTone` / `compileDrumKit` + `Instrument` from `wasy` to construct and route a live patch through Web Audio. New workspace package `wasy-patch-editor`.
- **Patch definition exports from `src/index.ts`**: `Instrument`, `Patch`, `Monophony` (class / type), `compileTone`, `compileDrumKit`, and the full `synth/types.ts` type set (`ToneDefinition`, `DrumKitDefinition`, `ToneSource`, `OscillatorSource`, `NoiseSource`, `PatchDefinition`, `AdsrEnvelope`, `RampEnvelope`, `Envelope`, `FrequencySpec`, `DrumVoiceDefinition`, `DrumRouting`) are now re-exported from the package entry. Applications can compile and play custom patches without reaching into internal module paths.
- **`instrumentPatchs` and `percussionKeyMap`** (from `src/midi/gm.ts`) re-exported from `src/index.ts`. Exposes the GM 128 instrument name list and the standard percussion key map for UI labeling (used by `examples/patch-editor` drum voice list).
- **Theme switcher in `examples/seekable-player/`** — new `src/theme.ts` manages `ThemeMode` (`"dark"` / `"light"` / `"system"`) and persists the selection to `localStorage`. `style.css` declares CSS custom properties (`--color-bg`, `--color-bg-alt`, `--color-border`, `--color-fg`, `--color-muted`, `--color-strong`, `--color-accent`, `--color-on-accent`, `--color-shadow`) for all three variants: Solarized Dark (`:root` default), Solarized Light explicit (`[data-theme="light"]`) and system-auto (`@media (prefers-color-scheme: light) :root:not([data-theme])`), Solarized Dark explicit (`[data-theme="dark"]`). Canvas views (`PianoRollView`, `KeyboardView`, `AnalyserView`) call `getCanvasPalette()` on every animation frame so redraws pick up the active theme without a reload. `palette.ts` extended with the full Solarized base-tone set: `base0` (#839496, dark-mode body text) and `base00` (#657b83, light-mode body text) added alongside the existing tones.
- **Channel Status View in `examples/seekable-player/`** — new `src/channel-status-view.ts`. A canvas strip with one cell per channel (0–15) showing the live active-voice count and the last received program number. NoteOn / NoteOff events increment / decrement the voice count; ProgramChange updates the program label. Channel 9 drum cells are tinted with neutral grey to match the piano roll and keyboard strip. Rendered to `<canvas>`; updates are batched per animation frame.

Selecting an entry fetches the file as `ArrayBuffer` and calls `player.load(buffer)` without a page reload. Falls back silently when `songs.json` returns an empty array.
- **Web MIDI panel in `examples/seekable-player/`** — new `src/web-midi-view.ts`. Provides: an **Enable Web MIDI** button (`navigator.requestMIDIAccess({ sysex: false })`); a live list of input ports with connected / disconnected state; force-program-change and channel-override controls for normalizing live input; and a **Force GM reset** button. Selected ports are forwarded to `engine.receiveExternalMidiEvent`. WebMidiLink (`window.message`) is always active.
- **MIT / Apache-2.0 dual license** — `LICENSE-MIT` and `LICENSE-APACHE` added to the repo root; `package.json` `"license"` updated to `"MIT OR Apache-2.0"`.
- **AHDSFR envelope (Attack / Hold / Decay / Sustain / Fade / Release).** The `Patch` base class envelope was previously plain ADSR; `Patch.applyAttack` now schedules two additional optional phases between Decay and Release: `holdTime` keeps the gain pinned at peak between the end of the attack and the start of the decay (mallet-style articulation — useful for bell / chime / tom-style patches), and `fadeTime` linearly ramps the (possibly already decayed) sustain level down to 0 over the given duration while the key is still held (useful for pads / voices / rain-style FX so they evolve rather than droning at a flat level). Setting either to 0 (or omitting from the JSON definition) reduces the schedule back to plain ADSR, so the change is backward-compatible with any caller that does not opt in. The schema-side type alias and discriminator tag stay named `AdsrEnvelope` / `"adsr"` — JSON literals authored against the prior schema continue to deserialize unchanged — but the type now also accepts optional `hold?` and `fade?` fields alongside `attack` / `decay?` / `sustain?` / `release`. Selected `gmPatches` voices were updated to demonstrate the new phases — bell-family voices (Glockenspiel / Vibraphone / Tubular Bell / Tinkle Bell) gained a brief peak `hold` for mallet articulation; Voice Aahs / Voice Oohs, Tremolo Strings, Pad 1..8, FX 1 (rain) / FX 2 (soundtrack) / FX 4 (atmosphere) / FX 5 (brightness) / FX 7 (echoes) / FX 8 (sci-fi), Reverse Cymbal, and Seashore gained a multi-second `fade` so held keys diminish naturally.
- **Data-driven patch definitions (`compileTone` / `compileDrumKit`).** The GM 128 melodic set and the Standard Drum Kit are now described as plain JSON-serializable objects (`ToneDefinition` / `DrumKitDefinition`) instead of being constructed inline by `if/else` branches and a hard-coded `patchMap` literal. New module layout: `src/synth/types.ts` (schema), `src/synth/compile.ts` (`compileTone` / `compileDrumKit` translate definitions into the existing `Patch` class hierarchy — audio path is unchanged), `src/synth/patches/gm.ts` (128-element `gmPatches: ToneDefinition[]`, each voice an explicit literal with `name` populated from `instrumentPatchs` — chiptune-flavored oscillator selection across sine / square / triangle / sawtooth / noise, all envelopes ADSR (now AHDSFR-extended; see the entry above) including the Piano family, which was previously a `ramp` overshoot+decay), `src/synth/patches/gm-drum-kit.ts` (`gmDrumKit: DrumKitDefinition` covering the full GM percussion key map 35..81 — the previous map only covered 35..51; new voices 52..81 add Chinese / Splash / Crash 2 / Ride 2, Ride Bell, Tambourine, Cowbell, Vibra Slap, bongos, congas, timbales, agogo, cabasa, maracas, whistles, guiros, claves, wood blocks, cuicas, and triangle, with name strings inlined per voice. SF2 exclude groups extended beyond hi-hats: short / long Whistle (group 2), short / long Guiro (group 3), mute / open Cuica (group 4), mute / open Triangle (group 5)), and `src/synth/generate-patch.ts` (the new `generatePatch` reads the data tables and dispatches to the compile functions). All five new modules are re-exported from `src/synth.ts` so the public surface is unchanged. `DrumKitPatch` gains a generalized `excludeGroups: Map<number, Set<number>>` (replacing the prior hard-coded `[42, 44, 46]` hi-hat exclusivity) — voices in the same group expire each other on attack (SF2 exclusive class semantics, with the struck voice itself never killed). `OneShotOscillatorPatch` constructor now accepts optional `valueAtBegin` / `valueAtEnd` (defaults `1` / `0` keep prior behavior) so ramp envelopes with non-default endpoints can compile through the same class.
- **Tempo map / time-signature map and tick→time / tick→bar:beat helpers in `smfAnalyze`.** `extractTempoMap(song)` and `extractTimeSignatureMap(song)` walk every track and return tick-sorted plain-data lists (`TempoChange { tick, microsecondsPerQuarter }` / `TimeSignatureChange { tick, numerator, denominator }`); both are now bundled into `SongInfo` (so the worker emits them once during init, alongside notes and metadata). `tickToSeconds(tick, tempoMap, resolution)` integrates piecewise across tempo segments (defaulting to 120 BPM / 500000 µs/quarter when the map is empty); `tickToBarBeat(tick, timeSignatureMap, resolution)` returns 1-indexed `{ bar, beat }` and starts a fresh bar at each TS change tick (defaulting to 4/4). `formatTime(seconds)` renders `mm:ss` (or `h:mm:ss` ≥ 1 hour). New `TimeSignatureMetaEvent` (typeIndex 0x58) class so consumers can `instanceof`-check time-signature events the same way as `TempoMetaEvent`. `examples/seekable-player` wires this into the seek-bar readout: live `mm:ss / mm:ss` time and `bar:beat`, plus a `mm:ss (NNN tick)` duration line in the metadata pane.
- **`smfAnalyze` namespace (`smfAnalyze.collectNotes` / `extractMetadata` / `computeDurationTicks` / `buildSongInfo`)** plus `Note` / `SongMetadata` / `SongInfo` types, all re-exported from the package entry. `collectNotes` walks all tracks, sorts by tick, and pairs NoteOn / NoteOff into closed `[startTick, endTick]` intervals (closing on same-pitch re-triggers and on the song's final tick for stragglers). `extractMetadata` pulls human-readable text meta-events into a single record (title from track-0's first `SequenceTrackName` per RP-019; copyright / track names / instrument names / markers / lyrics / generic text). `computeDurationTicks` returns the last event tick across all tracks. `buildSongInfo` packages everything (`format` / `numberOfTracks` / `resolution` / `durationTicks` / `notes` / `metadata`) into a plain-data object suitable for transfer through `postMessage`.
- **`SmfPlayer.songInfo: SongInfo | null` populated by the player worker on init.** The worker now runs `buildSongInfo()` once during SMF parsing and posts the result back as a `songInfo` message; `SmfPlayer` captures it, exposes it synchronously after `await load()`, and resolves the existing `ready` Promise on receipt (the redundant `resolution` round-trip is gone — `songInfo.resolution` carries the same value). UI consumers (piano-roll, metadata pane, mixer) no longer need to call `smf.parseSong(buffer)` on the main thread; the SMF is parsed exactly once, on the worker thread. `examples/seekable-player` now uses this directly: `notes.ts` and `metadata.ts` were deleted, and the main-thread parse on file load is gone (visible as a ~2 kB drop in the example's main bundle and a corresponding shift to the worker bundle).
- **GS Reset / XG System On SysEx detection.** `SynthEngine.receiveEvent` now matches the standard reset signatures (Roland GS Reset `F0 41 10 42 12 40 00 7F 00 41 F7`; Yamaha XG System On `F0 43 10 4C 00 00 7E 00 F7`) and calls a new `Instrument.applyReset(time)` on every part — which resets all controller state to GM defaults _and_ ramps the corresponding audio params back to those defaults (volume / expression / panpot / modulation / filter cutoff / filter Q / reverb / chorus / channel detune bus all use the same 8 ms anti-zipper ramp). CC 121 (Reset All Controllers) follows the narrower RP-015 scope via the separate `applyResetAllControllers` (see **Fixed**); the full GM-defaults reset is reserved for the System On SysEx path. Detection works both for SMF SysEx (which carries a leading variable-length count byte) and for runtime Web MIDI SysEx (which doesn't) — the matcher tries both offsets. Exported `matchSysEx` / `isGsReset` / `isXgReset` for callers that want to do their own SysEx triage.
- **Engine-level effect bus (Reverb + Chorus) and per-channel sends.** `SynthEngine` now owns a stereo `ConvolverNode` reverb (1.5 s exponentially-decaying noise IR, decay 2.0 — ~~"湿った" small-room feel~~) and a single-tap LFO chorus (`DelayNode` 25 ms ±5 ms at 0.5 Hz with 0.2 feedback). Both fan into the master `gain` so dry and wet share the same `DynamicsCompressor`. Per-channel send level is owned by each `Instrument` (`reverbSend` / `chorusSend` GainNodes; defaults follow GM2 — reverb send 40, chorus 0); `SynthEngine` taps `channelGain → instrument.reverbSend → engineReverb.input` (same for chorus) so user-gain (mute / solo / fader) and MIDI volume / expression apply to wet as well as dry.
- **CC 1 (Modulation Wheel)** drives a per-`Instrument` sine LFO whose depth (in cents) scales with the CC value. The LFO output sums into the channel detune bus connection, so vibrato propagates to all in-flight notes through the same per-note `oscillator.detune` / `filter.detune` connection that already carries pitch bend and RPN tuning.
- **GM2 vibrato + channel-mode controllers.** **CC 76 (Vibrato Rate)** sets the LFO frequency (64 = base 5 Hz; each ±64 steps is ∓ / ± one octave, so 0 → 2.5 Hz and 127 → ~9.9 Hz). **CC 77 (Vibrato Depth)** adds to the Mod Wheel's vibrato (its GM2 default 64 contributes nothing, so Mod-Wheel-only channels are unchanged; either control can drive vibrato on its own). **RPN 5 (Modulation Depth Range)** sets the full-scale vibrato amplitude (`modDepthRangeCents`, default 50 = 0.5 semitone; MSB = semitones, LSB = cents/128) that the Mod Wheel / CC 77 map onto — replacing the previously hard-coded ±50 cents. **CC 123 (All Notes Off)** and **CC 124–127 (Omni Off / On, Mono / Poly On, which each imply All Notes Off)** release every sounding note with a normal NoteOff via `Instrument.allNotesOff` — keeping the release tail, unlike CC 120 (All Sound Off) — and lift the sustain pedal so pedal-held notes release too. **CC 122 (Local Control)** is accepted as a no-op (a soft synth has no local keyboard to disconnect); Mono / Poly mode itself stays poly (a mono voice allocator is out of scope). The vibrato controllers are GM2 "sound controllers": preserved by CC 121 (RP-015) and reset only by GS / XG reset.
- **CC 71 (Filter Resonance)** maps to the per-`Instrument` `BiquadFilterNode.Q` over `[0.5, 12]` (logarithmic, 64 ≈ 1).
- **CC 74 (Brightness / Filter Cutoff)** maps to the per-`Instrument` `BiquadFilterNode.frequency` over `[750 Hz, 16 kHz]` (exponential, ~4.4 octaves; 64 ≈ 5.4 kHz). One filter per channel sits between `_panner` and `_gain`, defaulting to a near-bypass lowpass (12 kHz, Q=1) so patches without their own filter still respond to brightness changes without coloring at neutral CC values.
- **CC 91 (Reverb Send) / CC 93 (Chorus Send)** write the smoothed wet send level (0..127 → 0..1) onto each channel's per-`Instrument` send GainNode; the engine-level Reverb / Chorus do the rest.
  All five new CCs use the same 8 ms anti-zipper ramp as CC 7 / 10 / 11 and are reset to GM defaults by GS / XG reset (`applyReset`); CC 121 follows the narrower RP-015 scope and leaves them untouched (modulation excepted).
- **`examples/seekable-player` floating panel workspace.** Each section is now a `position: fixed` draggable panel with a title bar (the repurposed `<h2>`) and a close (×) button. Grabbing the title bar moves the panel via mouse and touch events (`touchmove` calls `preventDefault()` to suppress scroll); a `mousedown` anywhere inside the panel brings it to the front by incrementing a module-level z-index counter. A fixed `#view-bar` at the top of the viewport carries the app title and one checkbox per panel for toggling visibility from a single location; re-enabling a hidden panel resets its position to the configured default before showing it (recovery when a panel has been dragged off-screen). Default layout: left column (`x=5`) holds SMF loader / transport / waveform–spectrum; right column (`x=680`) holds piano-roll / keyboard. Mixer, event-log, and footnote panels start hidden and can be summoned from the view bar. All logic is encapsulated in a new `src/panels.ts` module (`initPanels()` called from `main.ts` on `DOMContentLoaded`).
- `examples/seekable-player/` event log view: a scrolling list of MIDI events arriving via `SmfPlayer.onTimedEvent`, rendered with channel-color coding, per-category checkboxes (Note / CC / Bend / Program / Pressure / Meta / SysEx), and pause / clear buttons. Decodes typed `MetaEvent` subclasses (Tempo → BPM; text Meta → decoded text) and names the common CCs (Volume / Sustain / RPN / Bank Select etc.) inline. Bounded ring buffer of 200 events; updates batched per animation frame to stay cheap during dense passages. Useful when investigating which CC or Meta event made the synth go off the rails.
- `examples/seekable-player/`: a Vite-based example that wires `SmfPlayer` + `SynthEngine` directly (no `Wasy` façade) and exposes a tick-precise seek bar. Demonstrates the parser / synth split: the app code calls `smf.parseSong()` on the main thread to read metadata (format / track count / resolution / duration) and to pre-extract paired NoteOn / NoteOff ranges for piano-roll rendering, constructs `SmfPlayer` for playback timing and `SynthEngine` for synthesis, and itself bridges them by forwarding `TimedEvent` to `engine.receiveEvent` at `timeStamp.accurateTime(event.tick)`. Includes a scrolling piano-roll visualization (channel-colored notes, beat / octave grid, now line, channel-9 in neutral grey for drums) that auto-fits its pitch range to the loaded SMF and previews the seek target while the slider is being dragged. Below the piano roll, a 16-row × 128-key keyboard activity strip (driven by `onTimedEvent` NoteOn / NoteOff, sharing the piano-roll palette) and a waveform / spectrum canvas (an `AnalyserNode` is inserted between `SynthEngine` and `audioContext.destination`). The metadata block surfaces title / copyright (from text Meta events) and a collapsible details view of track names, instrument names, markers and lyrics. A 16-channel mixer with master fader provides per-channel solo / mute / volume controls written to `SynthEngine.channelGains[ch].gain` and `SynthEngine.gain.gain` — independent of MIDI-driven CC 7 / CC 11.
- `SynthEngine.prewarm(time?)` and `Wasy.prewarm(time?)`: fire an inaudible velocity-1 NoteOn/NoteOff pair on all 16 channels at `time` (defaults to `audioContext.currentTime`). Allocates and connects the per-note Web Audio nodes (oscillator / filter / gain / buffer source) ahead of real playback so the first attack at song start doesn't pay graph-construction cost on the audio thread — that hitch was previously most audible on tick-0 drum hits at the top of playback. Velocity 1 squared (`(1/127)²`) sits ~ -84 dB before master gain (0.1) and the dynamics compressor; well below audibility on any sane monitoring chain. Chosen over `seek(0)` because `SmfPlayer.seek(tick)` drops NoteOn/NoteOff from the skipped range, so a `seek(0)` would silently eat tick-0 NoteOns. `examples/seekable-player` calls `synth.prewarm()` once when the audio graph is constructed.
- `SynthEngine.channelGains: GainNode[16]`: per-channel user-gain bus that sits between each `Instrument` and the master `gain`. Apps can write to `channelGains[i].gain` for mute / solo / per-channel volume without fighting MIDI-driven CC 7 / CC 11, which are still handled inside `Instrument` and remain authoritative for MIDI playback semantics.
- Typed `MetaEvent` subclasses for the SMF text Meta events (typeIndex 0x01–0x07) and tempo (0x51): `TextMetaEvent`, `CopyrightMetaEvent`, `SequenceTrackNameMetaEvent`, `InstrumentNameMetaEvent`, `LyricMetaEvent`, `MarkerMetaEvent`, `CuePointMetaEvent`, `TempoMetaEvent`. `MetaEvent.create()` now dispatches on `typeIndex`, so callers can `instanceof`-test the meta type instead of branching on `typeIndex`.
- `MetaEvent.text(encoding?)` now auto-detects encoding when called without an argument: tries strict UTF-8 first and falls back to Shift_JIS on invalid bytes (Japanese SMFs are commonly Shift_JIS, and SMF predates UTF-8). Callers that know the encoding can still pass it explicitly to skip the probe.
- `SmfPlayer.load(buffer)` now returns `Promise<void>` that resolves once the worker has parsed the SMF and posted back its `resolution`. `SmfPlayer.ready: Promise<void>` exposes the same future for callers that already issued `load()`. `Wasy.load(buffer)` and `Wasy.ready` mirror this. Awaiting before `play()` prevents the race where queued `read` messages flush back with a stale `timeStamp.currentTime` and tick-0 events get scheduled at a past audio time (Web Audio fires `start(timeInPast)` immediately, which manifested as drum hits firing before the downbeat at the start of playback).
- `smf` and `isDrumChannel` are now re-exported from the package entry (`src/index.ts`), so applications can extract SMF metadata or check GM2 drum-mode rules without reaching into internal paths.
- `Instrument` now handles **Sustain (Damper) Pedal (CC 64)**. While the pedal is held (value ≥ 64), incoming NoteOff events are deferred into a per-note map; on release they are dispatched in a single batch at the release-time. Re-pressing a sustained note (NoteOn arriving while the deferred NoteOff is still pending) cancels the pending release so the new attack supersedes it. AllSoundOff (CC 120), `pause()`, and `destroy()` clear the deferred map without dispatching (the notes themselves are killed); ResetAllControl (CC 121) and GS / XG reset dispatch the deferred NoteOffs first — treating the reset as a pedal release — so no notes are left stuck on.
- `synthEngine.isDrumChannel(channel, bankMSB)` exported from `synth-engine.ts`, encoding the GM2 / GS bank-driven drum-mode rule: bank MSB `0x78` forces drum on any channel, `0x79` forces melody (overriding the channel-9 default), and other bank MSBs fall back to "channel 9 = drum". `SynthEngine` now consults `isDrumChannel` both at instrument construction and on every ProgramChange, so a Bank Select followed by a Program Change can switch a channel between drum kit and melody patches.
- `SmfPlayer` (SMF playback: `Timer` + worker + tempo bookkeeping, emits `TimedEvent`) and `SynthEngine` (16 channels + master gain + dynamics compressor + MIDI routing) are now exported alongside `Wasy`. `Wasy` is a thin façade that wires `SmfPlayer` into `SynthEngine`.
- `Wasy.load(buffer)` and `Wasy.unload()` for swapping the loaded SMF without reconstructing the synthesizer.
- `Wasy.seek(tick)` and `SmfPlayer.seek(tick)` for jumping to an arbitrary tick. State events (ProgramChange / ControlChange / PitchBend / Tempo / SysEx) at ticks before the target are replayed so the synth state matches what it would have been at that tick; NoteOn / NoteOff events from the skipped region are dropped so notes do not retrigger.
- `generatePatch` now covers all 128 GM programs via a per-category fallback (8 programs per category × 16 categories). Unmapped programs pick a category-appropriate `OscillatorType` (e.g. strings → sawtooth, pad → triangle, reed → square) instead of always defaulting to `square`.
- `signal.createSignal<T>()` factory and `Signal<T>` interface (replaces the previous `Signal` class).
- `xiff.parseChunks(buffer, config)` (replaces `xiff.Chunk` class with a plain interface + parser function).
- `smf.parseHeader` / `smf.parseTrack` / `smf.parseSong` (replaces `Header` / `Track` / `Song` classes).
- `player.createPlayer(buffer)` (replaces the `Player` class).
- `synth.generatePatch(instrument, program, isDrum)` (replaces the `PatchGenerator` class).
- `webmidi.createWebMidiInput()` / `createWebMidiLinkInput()` (replaces the `MIDIIn` / `WebMIDIIn` / `WebMidiLinkIn` class hierarchy).

### Changed

- **`examples/seekable-player` drag & drop moved to the Load SMF panel.** Dropping a file used to target the Piano Roll; it now targets the **Load SMF** panel (with a dashed drop-zone highlight) and reflects the dropped file into the `<input type="file">` so its filename label matches what's playing. A window-level handler swallows the browser's default "open the file / navigate away" on drops outside the panel. Transport / action buttons are Title Case (Play / Pause / Stop / Resume / Clear).
- **`examples/seekable-player` UI copy pass.** Axis hints now read in conventional X-then-Y order ("X = time, Y = pitch" for the piano roll; "X = key number, Y = MIDI channel" for Channel Notes) and the piano-roll cursor is called the "playhead". The analyser legend spells out spectrum / waveform; tick readouts are pluralized ("0 / 0 ticks"); the metadata resolution label reads "ticks/quarter" instead of the jargon "TPQN"; Web MIDI status lines share one "Web MIDI …" phrasing; prose spells out "Standard MIDI File" where the compact "SMF" label isn't needed. The Preferences lookahead hint was corrected — it previously claimed the views lead the sound by the lookahead, which is no longer true since they track `audibleTick`; it now describes the real trade-off (jitter robustness vs. play/seek responsiveness).
- **`examples/seekable-player` mixer faders use a quadratic taper.** The master fader (0–100) now defaults to **80**, which reproduces the engine's reference master gain (0.1, the headroom-safe level for 16 summing channels); 100 gives ~+3.9 dB of boost and 0 is silence, via `gain = 0.1 × (v/80)²`. Previously the slider mapped linearly to gain 0–1, so the default sat at an awkward "10" and everything above ~30 was uncomfortably loud. Channel faders apply the same quadratic taper with unity at the default 100 (attenuate-only), matching the GM2 CC 7 / CC 11 curve the synth itself uses. The master fader position is persisted to `localStorage` (same approach as the Preferences lookahead) and restored on load.
- **Channel state (PitchBend / fine tune / coarse tune) now propagates to in-flight notes** via a per-`Instrument` `ConstantSourceNode` ("detune bus"). The bus emits the channel's total detune offset in cents (`pitchBend + fineTune + coarseTune × 100`) as a DC value; on NoteOn each patch routes the bus's `offset` into the new note's detunable AudioParams (`oscillator.detune` / `filter.detune`) via `connect`, and disconnects on the source's `ended` event. Writes to a single `_detuneOffset.offset` AudioParam (with the same 8 ms anti-zipper ramp used for volume / panpot) flow through the connection summed at audio rate, so a PitchBend or RPN 1 / 2 update reaches every sounding note instantly without iterating the NotePool. Per-note `oscillator.detune.value = this.detune` snapshots at NoteOn — which froze pitch bend / RPN tuning at attack time — are gone; oscillator / filter detune bases are now `0` and the bus contributes the offset. New `Patch.attachChannelDetune(monophony, source)` helper handles the connect/disconnect lifecycle. `Patch.onPitchBend` and the per-note `PitchBendEvent` dispatch in `Patch.receiveEvent` were removed (the event is now intercepted at the `Instrument` level). Subclasses no longer need a PitchBend handler.
- **`Patch` base class now provides an ADSR envelope** (`attackTime`, `decayTime`, `sustainLevel`, `releaseTime`) plus protected `applyAttack(gainParam, peakGain, time)` / `applyRelease(gainParam, time)` helpers. Defaults: 5 ms attack, 0 decay, sustain at peak, **50 ms release**. `SimpleOscillatorPatch` and `NoisePatch` now schedule attack via `applyAttack` and release via `applyRelease` (oscillator / source `stop()` is deferred to `time + releaseTime` so the release tail isn't cut off). This eliminates the sample-boundary clicks at NoteOff and gives Sustain Pedal releases an audible decay instead of an instant cut. `applyRelease` uses `cancelAndHold(param, time)` (from `src/synth/audio-param.ts`) so the release ramp anchors at the envelope's value at the _scheduled_ NoteOff time (matters for the player's ~200 ms lookahead) — on browsers without `cancelAndHoldAtTime` the value is computed from the shadow-tracked schedule (see **Fixed**). `GainedOscillatorPatch` / `GainedNoisePatch` (piano-style decay envelope and percussion one-shots) now compute `baseGain` from `velocityToGain(event.velocity)` directly instead of reading `gain.gain.value` after super, since the base attack now schedules events at audio `time` (so `.value` at currentTime is still 0); audible behavior is unchanged for those patch families. `OneShot*` patches keep their explicit gain-zero cleanup on expired (drum-sample stealing remains an instant cut, which their own one-shot ramp masks).
- **CC 7 (Volume) / CC 11 (Expression) now follow the GM2 response curve** `dB = 40·log10(v/127)`, i.e. gain `(v/127)²` per controller, replacing the previous linear `v/127`. This matches the squared velocity curve (below) and the GM2 spec; defaults get slightly quieter (volume 100 → gain ≈ 0.62 instead of 0.79). The channel gain node is also initialized to the GM default level (volume 100 × expression 127) instead of the raw 1.0, so channels that never receive CC 7 play at the spec-default loudness.
- **Default Reverb Send Level is now 40** (the GM2-specified default; chorus stays 0). Previously both defaulted to 0 (dry-only), so SMFs that never send CC 91 got no reverb at all. Files that do send CC 91 are unaffected.
- **Velocity → Gain curve is now squared (`(v/127)²`)**, not linear. The MIDI spec leaves the curve implementation-defined; DLS Level 1 / GM convention and most hardware GM modules use roughly `(v/127)²`. Linear `v/127` made soft notes louder than expected and the dynamic range felt compressed; squaring gives a more natural piano-like response. Applied uniformly to `SimpleOscillatorPatch` and `NoisePatch` (so all derived patches including `OneShot*`, `Gained*`, and `DrumKitPatch` slots inherit it).
- **CC 7 / CC 11 / CC 10 (volume / expression / panpot) are now anti-zipper smoothed** with an 8 ms `linearRampToValueAtTime`, replacing the previous `setValueAtTime` step that produced sample-boundary clicks on rapid CC sweeps. `setVolume(value, time)` and `setExpression(value, time)` route through a shared `_rampGain` helper; `setPanpot` now also takes a `time` parameter (previously a void-time direct `.value` write) and ramps `_panner.pan` the same way. Existing CC 7 / 10 / 11 dispatch in `receiveEvent` already had the audio time in scope, so no caller change is required. All channel ramps (volume / expression / panpot / modulation / filter cutoff & Q / reverb & chorus sends / detune bus) now share a single `_rampParam` helper that anchors via `cancelAndHold(param, time)` instead of `setValueAtTime(param.value, time)`, so back-to-back CC changes scheduled inside the lookahead window chain from the correct scheduled value rather than restarting from the stale `currentTime` value.
- `Wasy.onTimedEvent` now fires once per source MIDI event. Previously, non-channel events (Meta / SysEx) were emitted up to 16 times because the worker duplicates them across channel buckets; the deduplication now happens at the `SmfPlayer` boundary. Listeners that filtered for `ChannelEvent` are unaffected.
- TypeScript strict mode is now fully enabled, including `strictPropertyInitialization`, `noImplicitOverride`, `noUnusedLocals`, and `noUnusedParameters`.
- `Patch` is now an `abstract class` and `Patch.onNoteOn` returns `T` (non-null) — every concrete subclass already returned a `Monophony`, so the `T | null` contract was misleading.
- `Instrument` and `DrumKitPatch` panning use `StereoPannerNode` instead of the deprecated `PannerNode`.
- `NoisePatch` keeps a per-`AudioContext` noise buffer (via `WeakMap`) instead of a single static buffer, so multiple `AudioContext` instances no longer share a sample-rate-mismatched buffer.
- `Timer.start()` / `resume()` share a single private `_scheduleInterval()` helper.
- ESM-only output, `target: ES2022`, `module: NodeNext`. The bundled webpack 3 / ts-loader pipeline is gone; library builds via `tsc` and the example builds via Vite 6.
- `Timer` is now anchored to the audio clock ("Two Clocks" pattern). `timing()` advances `tick` by `ticksPerSecond * (audioContext.currentTime - this.currentTime)` — the actual elapsed audio time — instead of the previously-assumed `ticksPerSecond * durationInSeconds`. The default `durationInSeconds` is now 25ms (was 200ms), giving the scheduler 8× more frequent wake-ups while `delayInSeconds` (lookahead) stays at 200ms. As a result, `setInterval` jitter no longer slows the music down. `Timer.resume()` resets the anchor to `audioContext.currentTime` so the first post-pause fire doesn't see the whole pause as elapsed.

### Fixed

- **Late-dispatched events crushed release envelopes into clicks.** With a small scheduling lookahead, `setInterval` jitter can dispatch a NoteOff after its scheduled audio time. Web Audio clamps the past-time automation events to "now" but leaves ramp _endpoints_ where they were, so the 50 ms release ramp collapsed to near-zero (audible click / abrupt note end) and `oscillator.stop(time + release)` cut early — making note endings sound different depending on the lookahead setting. `SynthEngine.receiveEvent` now clamps the event time up to `audioContext.currentTime`, converting lateness into a small timing slip while keeping every envelope shape intact (clamping is monotone, so event order is preserved).
- **Shadow-tracker pruning could silence mid-attack releases on Firefox.** The `cancelAndHoldAtTime` fallback pruned the shadow event list down to 2 entries before each newly inserted event, assuming queries only move forward — but a NoteOff can query a time _earlier_ than already-scheduled decay / fade events. For AHDSFR voices with hold / fade phases (5–6 scheduled events), the attack segment got pruned and the release anchored at the stale `param.value` (0), silencing the note. The keep-count is now 8, which covers a full AHDSFR envelope while still bounding growth on long-lived channel params.
- **Loading a new SMF inherited the previous song's programs, controllers, and tempo.** `Wasy.load()` / `unload()` only killed active notes, so song B started with song A's program changes, volume / expression (a fade-out ending left the next song silent), pitch bend range, tuning, filter, sends — and, when song B had no tempo event, song A's last tempo instead of the SMF-default 120 BPM. New `SynthEngine.applyResetAll(time?)` resets all 16 parts to GM power-on defaults — `Instrument.applyReset` plus the program back to 0 (piano / Standard Kit on the drum part) — and is called from `Wasy.load()` / `unload()`; `SmfPlayer.unload()` now also resets the timer to 120 BPM. The GS Reset / XG System On SysEx path routes through `applyResetAll` too, so a mid-song GS / XG reset now restores default _programs_ in addition to controllers (matching SC-55 / MU-series behavior). `examples/seekable-player` calls `applyResetAll()` when loading a file.
- **CC 121 (Reset All Controllers) reset far more than RP-015 / GM2 allow.** It routed through the full GM-defaults reset (`applyReset`), wiping volume / panpot / bank select / effect sends / sound controllers and the RPN values (pitch bend range, fine / coarse tune) — so a mid-song CC 121 audibly destroyed the mix. It now calls the new `Instrument.applyResetAllControllers(time)`, which implements the RP-015 scope GM1 / GM2 mandate: modulation → 0, expression → 127, sustain → off (releasing pedal-held notes), pitch bend → center, and RPN / NRPN _selection_ → null. The full reset remains available as `applyReset` and is only triggered by GS Reset / XG System On SysEx.
- **Resets dropped pedal-held NoteOffs, leaving notes stuck on.** `resetAllControl` cleared `_sustainedNoteOffs` without dispatching, so notes held by the sustain pedal at the moment of a CC 121 / GS / XG reset never received their NoteOff and rang until polyphony eviction. Both reset paths now run `setSustain(0, time)` first, dispatching the deferred NoteOffs at the reset time like a real pedal release.
- **CC 120 (All Sound Off) ignored the event's audio time.** `notePool.unregisterAll()` was called without the time argument (defaulting to 0 = "as soon as possible"), cutting sound ~200 ms early under the player's lookahead. The CC's scheduled time now flows through to the expire notifications.
- **Release / one-shot anchoring on browsers without `cancelAndHoldAtTime` (Firefox, Bugzilla 1308431).** The fallback used to anchor at `param.value` — the value at `currentTime` — which diverges from the scheduled anchor `time` under the player's ~200 ms lookahead. Worst case: a NoteOff scheduled while the 5 ms attack ramp had not yet started in real time read `param.value` as 0 and anchored the release there, silencing the whole note. The new `src/synth/audio-param.ts` layer shadow-tracks scheduled events (`setValueAtTime` / `linearRampToValueAtTime` are the only primitives the codebase uses) and computes the exact automation value at `time` by linear interpolation. All envelope scheduling (`applyAttack` / `applyRelease`, the `Gained*` decay ramps, the `OneShot*` expire cleanup) now routes through the helpers so the shadow list stays in sync. (Tracking is unconditional on every browser — see the next entry for why the native `cancelAndHoldAtTime` is deliberately not used.)
- **Releasing a sustained note on Chrome / Safari started the fade a whole lookahead early.** Native `cancelAndHoldAtTime` inserts a hold event only when it truncates something spanning the cancel time; for a sustained note the entire envelope (the attack ramp) lies _before_ the NoteOff, so nothing was inserted — and the subsequent `linearRampToValueAtTime(0, time + release)` anchored at the **attack end** per spec ("changes start at the time of the previous event"). The gain visibly dropped the moment the ramp was inserted, i.e. at dispatch time, `lookaheadSeconds` before the musical NoteOff — clearly audible at 500 ms, subtle but present at the 200 ms default. Firefox (which lacks the method and used the shadow fallback) was unaffected, which is what isolated the bug. `cancelAndHold` now always pins the computed value with an explicit anchor at the hold time on every browser, so following ramps start exactly there.
- **Expiring a decaying drum voice made it swell back to peak for a whole lookahead (cymbals "starting early").** `cancelAndHold`'s `cancelScheduledValues(time)` removes an in-flight decay ramp _entirely_ (its event time is the ramp end), so from the dispatch moment until `time` the param reverted to the previous event's value — the full peak. A crash cymbal expired by the next hit (same-note retrigger, exclude group, or LRU eviction) audibly jumped back to full volume `lookaheadSeconds` before that hit, sounding like the cymbal struck early; the longer the lookahead, the longer the resurrected peak. This is the historic swell-back hazard the native `cancelAndHoldAtTime` used to avoid by _truncating_ (not removing) the spanning ramp — dropping the native call reintroduced it on Chrome / Safari. The anchor is therefore a **`linearRampToValueAtTime(value, time)`** rather than a `setValueAtTime`: ramping from the previous event to `(time, value)` re-traces the cancelled linear decay exactly (the held value lies on the original line), degenerates to a flat hold for sustained releases, and still gives later ramps their starting point. One mechanism now covers both the sustained-release anchoring and in-flight-decay truncation on every browser.
- **`createWebMidiLinkInput` threw on non-string `MessageEvent.data`** — `window.message` listeners receive events from HMR servers, DevTools extensions, and other frame sources that post non-string payloads. The previous code called `event.data.split(",")` unconditionally, which threw a TypeError on non-string values. The handler now returns early when `typeof event.data !== "string"`.
- **`gmPatches` brass / woodwind voices missing `hold` phase** — Trumpet, Trombone, Tuba, French Horn, Brass Section, Synth Brass 1–2, and several adjacent woodwind voices lacked a peak hold between attack and decay; the result was an immediate drop to the sustain level right after the attack peak, unlike the expected brass articulation. Added `hold: 0.1` (100 ms) to the affected voices.
- **Solarized Dark `--color-fg` rendered body text in accent red** — the `:root` (default dark-theme) block had `--color-fg: #dc322f` (Solarized red), which is the same value as `--color-accent`, causing all body text to appear in the interactive/accent hue. Corrected to `#839496` (Solarized `base0`, the canonical dark-mode body text tone per the Solarized specification).
- **`OneShotNoisePatch.onExpired` / `OneShotOscillatorPatch.onExpired` no longer make long-decay drums (e.g. Crash Cymbal 1, `OneShotNoisePatch(is, 1, 0, 1.5, 8000, ga)`) audibly swell right before a re-trigger.** The previous `cancelScheduledValues(time) + setValueAtTime(0, time)` cleanup, combined with the player's 200 ms lookahead, left only the initial `setValueAtTime(peak, t0)` event in the schedule for the `[currentTime, time]` window — so the gain held at peak for that whole window before snapping to 0, making the previous hit ring louder right before the new attack and shifting the perceived timing of the new hit ~200 ms early. The expired path now anchors the in-flight decay ramp at `time` via `cancelAndHold(param, time)` from `src/synth/audio-param.ts` (native `cancelAndHoldAtTime` where available, shadow-tracked interpolation on Firefox — see the dedicated entry above), then schedules a 5 ms `linearRampToValueAtTime(0)` and defers `source.stop` / `oscillator.stop` to `time + 0.005` so the brief fade isn't cut off.
- `Event.create`: NoteOn with velocity 0 is now converted to a NoteOff that preserves the original channel (previously the channel bits were dropped).
- `Instrument.receiveRPN`: DataEntry MSB / LSB were swapped, and an LSB-only update never fired `receiveRPN`. Both paths now follow the standard `(MSB << 7) | LSB` encoding, and pitch bend range RPN 0 is interpreted as `MSB semitones + LSB cents / 100`.
- `Instrument` now tracks Bank Select (CC 0 / 32) and NRPN (CC 98 / 99) state. DataEntry (CC 6 / 38) routes to `receiveRPN` or the new `receiveNRPN` hook based on whichever parameter pair was selected most recently. RPN / NRPN initial value is 0x3FFF (null) per spec, so a stray DataEntry no longer accidentally writes pitch bend range.
- `Instrument.receiveRPN` now handles RPN 1 (Channel Fine Tuning, 14-bit, ±100 cents) and RPN 2 (Channel Coarse Tuning, MSB-only, ±64 semitones). The new `fineTune` / `coarseTune` fields fold into `Instrument.detune` so any pitch-bend-driven node retune sees the combined offset. Together with the new channel detune bus (see **Changed**), in-flight notes also pick up RPN tuning changes immediately.
- `Wasy.destroy()` now terminates the player Web Worker (was leaked previously).
- `binary/dataViewGetUint`: switched from `<<` shifts (which break past 32 bits) to multiplication-based accumulation, accurate up to ~6 bytes / `Number.MAX_SAFE_INTEGER`.
- `WebMIDIIn` (now `createWebMidiInput`): leftover `console.log(input)` debug output removed.
- SMF `ControlChange` length: confirmed all CC messages (including Channel Mode 0x7C / 0x7E) consume 2 data bytes; stale FIXME comment removed.
- SMF Format 2 input now logs a `console.warn` rather than silently parsing under unspecified semantics.

### Removed
